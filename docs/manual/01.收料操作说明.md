# 入库收料操作说明

> 适用端：PDA（主） / PC Web（辅）
> 对应角色：仓库操作员
> 最后更新：2026-06-18
> 关联版本：v1.0

---

## 一、业务目标

将**客户来料**录入系统库存，使其成为可追溯的库存盘（Reel），并为后续上架/发料做准备。

收料环节完成三件事：
1. **识别物料** — 通过扫描客户标签，自动或人工确认物料编码
2. **创建库存** — 生成系统库存盘记录（InventoryReel）
3. **生成内部标签** — 打印含内部料号、客户料号、Reel 编号、数量的标签贴于盘上

---

## 二、操作流程

### 2.1 前置条件

| 条件 | 说明 |
|------|------|
| 客户标签完好 | 来料外箱/盘上附有条形码或二维码标签 |
| 打印机已连接 | 标签打印机网络可达（IP + Port 已在系统设置中配置） |
| 物料已建档 | 客户料号已在系统物料主数据或客户物料映射中存在 |

### 2.2 PDA 端操作（推荐）

PDA 是收料的主要操作端，流程简洁：

#### Step 1：开始入库

| 操作 | 说明 |
|------|------|
| 首页 → 入库 | 进入入库收料首页 |
| 输入操作员姓名 | 在输入框中填写操作员姓名 |
| 点击"开始入库" | 系统自动创建入库单（编号格式 `RC-YYYYMMDD-001`，每日从 001 开始顺序编号），进入扫码界面 |

#### Step 2：扫码收料（核心步骤）

进入扫码界面后，输入框自动聚焦，扫描客户标签上的条码即可。

**系统自动执行以下流程**：

```
扫描条码
   │
   ▼
① 重复检查 ── 已在库？ ──→ 根据配置决定：拦截 / 警告 / 放行
   │ 未重复
   ▼
② 客户物料映射查找 ── 精确匹配？ ──→ 自动入库（置信度 1.0）
   │ 无映射
   ▼
③ 条码解析 + 模糊匹配 ── 最高置信度 ≥ 60%？ ──→ 自动入库
   │ 置信度 30%~60%
   ▼
④ 返回候选物料列表（TOP 5）── 需人工在 PC Web 上确认
   │ 无候选
   ▼
⑤ 自动创建新物料 ──→ 入库完成
```

#### Step 3：扫码结果处理

系统返回结果后，根据状态不同采取不同操作：

| 状态 | 显示样式 | 说明 | 操作员动作 |
|------|---------|------|-----------|
| ✅ 成功 (ok) | 绿色 | 自动匹配成功，库存盘已创建，标签已打印 | 贴标 → 继续扫下一盘 |
| ⚠️ 需确认 (pending_review) | 黄色 | 匹配置信度 30%~60%，返回候选列表 | 在 **PC Web** 上确认物料（见 2.3 节） |
| 🆕 新料 (new_material) | 黄色 | 无匹配候选，自动创建新物料 | 确认物料信息是否正确 → 继续 |
| 🔁 重复 (duplicate) | 黄色 | 该条码已在库存中存在 | 检查是否重复操作 |
| ❌ 错误 (error) | 黄色 | 条码格式无效或系统异常 | 检查条码 → 重扫 |

#### Step 4：结束入库

PDA 端无"确认入库单"或"完成入库单"操作。扫码结束后退出入库界面即可。

如需完成入库单，请在 **PC Web 端**操作（见 2.3 节）。

---

### 2.3 PC Web 端操作

PC Web 端适合批量收料、物料确认、入库单管理等场景。

#### Step 1：创建收料单

1. 进入「入库收料」页面
2. 点击「新建收料单」
3. 输入操作员姓名 → 确认
4. 系统生成入库单号 `RC-YYYYMMDD-001`，状态为 `draft`

#### Step 2：扫码收料（两步流程）

PC Web 端扫码采用 **预览 + 确认** 的两步流程，方便核对物料信息。

**方式一：自动匹配（两步合一）**

| 操作 | 系统行为 |
|------|---------|
| 点击"扫码入库" → 扫描条码 | 调用预览接口 → 自动匹配成功 → 自动确认入库 |

适用于置信度 ≥ 60% 或客户物料映射精确匹配的场景。

**方式二：人工确认（预览 + 确认）**

当匹配置信度 < 60% 或为新料时：

1. 扫描条码后，系统返回预览卡片，显示：
   - 条码解析出的物料编码
   - 候选物料列表（TOP 5，按置信度排序）
   - 可编辑字段：数量、批次号、Date Code、规格
2. 从候选列表中选择已有物料，或选择"新建料号"
3. 如需新建，填写物料编码和名称
4. 点击确认 → 系统执行入库

#### Step 3：管理收料单

在「入库收料」页面的收料单列表中：

| 操作 | 按钮 | 说明 |
|------|------|------|
| 查看详情 | 查看 | 弹出详情弹窗，显示已收物料明细（物料编码、Reel#、条码、标签打印状态） |
| 确认收料 | 确认 | 状态 `draft` → `confirmed`，可继续补扫 |
| 完成收料 | 完成 | 状态 `confirmed` → `completed`，不可再扫码 |
| 补打标签 | 补打标签 | 打开弹窗，选择已收物料逐项补打内部标签 |
| 分配储位 | 分配储位 | 手动为指定明细行分配储位（见上架操作说明） |

#### 收料单状态流转

```
  draft ──→ confirmed ──→ completed
    ↑            │
    └────────────┘  (draft 可多次确认)
```

- **draft**：可继续扫码入库
- **confirmed**：已确认，但仍可补扫（后续也可继续补扫确认）
- **completed**：已完成，锁定，不可再扫码

### 2.4 扫码场景详细说明

#### 场景 A：自动匹配成功 ✅

**条件**：客户物料映射精确匹配（置信度 1.0）或模糊匹配置信度 ≥ 60%。

**系统行为**：
1. 自动创建库存盘（InventoryReel），状态 = `on_shelf`
2. 自动分配空储位（若有空闲储位）
3. 自动触发标签打印
4. 记录入库交易流水

**结果卡片**：
```
┌─────────────────────────┐
│  状态: ✅ 成功            │
│  条码: 2024ABC-12345     │
│  物料编码: RES-0402-1001 │
│  物料名称: 贴片电阻 100Ω  │
│  Reel #: 1024            │
│  分配储位: SH-A-01-05    │
│  数量: 1 盘              │
│  标签: 已打印            │
└─────────────────────────┘
```

**操作员**：将打印出的标签贴到盘上 → 继续下一盘。

#### 场景 B：匹配置信度低，需人工选择 ⚠️

**触发条件**：模糊匹配置信度在 30%~60% 之间，无精确匹配。

**系统行为**：
- PDA 端：返回黄色警告状态，附带候选物料列表
- PC Web 端：弹窗显示候选物料，操作员需手动选择

**操作员**：
- **有匹配** → 选择对应物料 → 确认入库
- **无匹配** → 选择"新建料号" → 填写物料编码和名称 → 确认

> 新建物料后，系统自动创建 `MaterialMaster` 记录，unit 默认为"盘"。

#### 场景 C：完全不匹配，作为新料入库 🆕

**触发条件**：模糊匹配置信度 < 30%，无任何候选物料。

**系统行为**：
- 自动创建新物料记录
- 物料编码默认使用从条码中提取的编码，可修改
- 创建库存盘并完成入库

**操作员**：确认物料信息 → 继续。

#### 场景 D：重复扫码 🔄

**触发条件**：扫描的条码已在库存中存在（status = `on_shelf` 或 `tracking`）。

**系统行为**：取决于系统设置 `duplicate_scan_behavior`：

| 配置值 | 行为 | 说明 |
|--------|------|------|
| `block`（默认） | 拦截扫描，返回重复提示 | 防止重复入库 |
| `warn` | 允许入库，标记 `duplicate_flag=true` | 放行但标注重复 |
| `force` | 完全跳过检查，当作新料入库 | 不检查重复 |

**操作员**：检查是否重复操作。

## 三、数据流参考

### 3.1 涉及的数据库表

| 表名 | 操作 | 说明 |
|------|------|------|
| `receipt` | INSERT | 入库单头（自动编号 RC-YYYYMMDD-NNN） |
| `receipt_reels` | INSERT | 每盘收料记录（关联入库单 + 库存盘） |
| `inventory_reels` | INSERT | 库存盘记录（物理盘） |
| `inventory_reels.shelf_slot_id` | UPDATE | 自动分配储位时写入 |
| `material_master` | INSERT | 仅当确认为新料时自动创建 |
| `customer_material_mappings` | SELECT | 客户料号 → 内部料号映射（优先级最高） |
| `transactions` | INSERT | 自动记录入库交易流水（type=in） |
| `shelf_slots` | SELECT | 查询空闲储位 |

### 3.2 物料匹配置信度阈值

| 阈值 | 状态 | 行为 |
|------|------|------|
| 1.0（精确） | `ok` | 客户物料映射命中，自动入库 |
| ≥ 0.6 | `ok` | 模糊匹配置信度高，自动入库 |
| 0.3 ~ 0.6 | `pending_review` | 返回候选列表，需人工确认 |
| < 0.3（无候选） | `new_material` | 自动创建新物料 |

### 3.3 入库单状态机

```
  draft ──→ confirmed ──→ completed
    ↑            │
    └────────────┘  (可多次补扫确认)
```

### 3.4 库存盘状态机

```
  on_shelf ──→ in_use ──→ tracking ──→ exhausted
  (刚入库)    (发料中)   (退库/待补)  (数量=0)
```

## 四、条码解析能力

系统内置条码解析引擎，支持以下格式：

| 格式 | 说明 |
|------|------|
| 标准组件码 | RES、CAP、IND、IC 等前缀的物料编码 |
| GS1-128 / EAN-128 | 含 Application Identifiers (AI) 的标准条码 |
| 供应商 2D Data Matrix | Murata、TDK、Samsung 等品牌格式 |
| 多分隔符支持 | `~`、`^`、`|`、`/`、`;`、`,` |

**解析提取字段**：
- 物料编码
- 数量（QTY、GS1 AI 30/37 等）
- 批次号（Lot/Batch，GS1 AI 10）
- Date Code（GS1 AI 11/17、D/C: 模式）
- 规格（SMT 封装尺寸，如 0402、0603、0805 等）
- 供应商编码（GS1 AI 240 或 GTIN）

## 五、标签打印

### 打印流程

- 每次扫码入库成功后，系统自动通过 TCP（ZPL 协议，默认端口 9100）发送标签打印指令
- 标签内容包含：
  ```
  内部料号: RES-0402-1001
  客户料号: CUST-ABC-12345
  Reel: #1024
  QTY: 3000
  ════════════════════════════
  RES-0402-1001
  ```

### 补打标签

- 若标签未打印（打印机离线等），可在 PC Web 端「入库收料」页面点击「补打标签」
- 弹窗中选择已入库的物料，点击对应行的补打按钮重新发送打印指令
- 补打成功后，`internal_label_printed` 标志更新为 1

## 六、异常处理

| 异常 | 系统行为 | 操作员处理 |
|------|---------|-----------|
| 打印机离线 | 入库成功，标签标记未打印 | 稍后在 PC Web 端补打标签 |
| 条码无效 | 返回"无效的条码格式"错误 | 检查条码是否完整，重扫 |
| 储位已满 | 入库成功，不分配储位 | 后续在上架环节手动分配 |
| 网络断开 | PDA 本地缓存，上线后自动提交 | 确认缓存队列正常 |
| 重复扫同一盘 | 根据配置拦截/警告/放行 | 确认是否重复操作 |
| 物料匹配失败 | 自动创建新物料 | 确认物料信息，后续补充完善 |

## 七、常见问题

**Q: 为什么扫了条码提示"需确认"？**
A: 表示匹配置信度在 30%~60% 之间，系统不确定对应哪个物料。请在 PC Web 端查看候选列表并手动选择正确的物料。

**Q: 标签没打印出来怎么办？**
A: 检查打印机 IP/Port 是否在系统设置中正确配置。入库本身不受影响，可在 PC Web 端「入库收料」页面点击"补打标签"手动补打。

**Q: 扫码时选错了物料怎么办？**
A: 当前不支持直接撤销。可在库存管理中删除刚创建的库存盘（需管理员权限），然后重新扫码。

**Q: 一个收料单可以扫多少盘？**
A: 不限。可在"完成收料"前持续扫码，所有扫码记录归入同一收料单。

**Q: PDA 端和 PC Web 端有什么区别？**
A: 
- PDA 端：扫码为主，一步到位，适合产线/仓库现场快速收料
- PC Web 端：预览 + 确认两步流程，适合批量处理、物料确认、入库单管理
- 两端共用同一套后端逻辑，数据实时同步

**Q: 如何配置重复扫码的处理方式？**
A: 在「系统设置」中修改 `duplicate_scan_behavior` 参数：`block`（拦截）、`warn`（警告）、`force`（强制放行）。
